# Database Schema & Isolation

## Data Architecture

The system employs a **Database-per-Tenant (Isolated Schema)** strategy locally achieved via distinct SQLite database files (e.g., `databases/tenant_<uuid>.sqlite`), managed by the Master Database.

## Master Database (`database.sqlite`)

The master database orchestrates the multi-tenant architecture and cross-system data.

### 1. `tenants`

* `id` (UUID - PK)
* `name` (String - Store name)
* `cnpj` (String - Unique)
* `status` (Enum - ACTIVE, SUSPENDED, INACTIVE)
* `email` (String)

### 2. `plans`

* `id` (UUID - PK)
* `name` (String - e.g., 'Free', 'Starter')
* `price` (Decimal)
* `osLimit` (Int - Maximum monthly OS. 0 = unlimited)
* `usersLimit` (Int)

### 3. `subscriptions`

* `id` (UUID - PK)
* `tenantId` (UUID - FK to Tenants)
* `planId` (UUID - FK to Plans)
* `status` (Enum - TRIAL, ACTIVE, PAST_DUE, CANCELLED)
* `nextBilling` (DateTime)

### 4. `users`

* `id` (UUID - PK)
* `tenantId` (UUID - FK to Tenants, Indexed)
* `email` (String - Unique)
* `password` (String - Hashed)
* `role` (Enum - Admin, Technician, Attendant, Super_Admin)

## Tenant Database (`tenant_<id>.sqlite`)

Dynamically generated when a tenant is registered. Contains operational data completely siloed from other stores.

### 1. `clients`

* `id` (UUID - PK)
* `nome` (String)
* `documento` (String)
* `status` (String)

### 2. `order_services`

* `id` (UUID - PK)
* `protocol` (String - Auto-generated sequence)
* `clientId` (UUID - FK to Clients)
* `status` (Enum - ABERTA, EM_DIAGNOSTICO, AGUARDANDO_APROVACAO...)
* `priority` (Enum)
* `tenantId` (UUID - Ensuring strict context)

### 3. `order_equipments`

* `id` (UUID - PK)
* `orderId` (UUID - FK to OrderServices)
* `type` (String - e.g., Smartphone, Laptop)
* `brand` / `model` / `serialNumber` (String)

### 4. `order_histories`

* `id` (UUID - PK)
* `orderId` (UUID - FK to OrderServices)
* `actionType` (Enum - SYSTEM, STATUS_CHANGE, INTEGRATION)
* `comments` (Text)
* `waMsgSent` (Boolean - Tracks WhatsApp notifications)

### 5. `products` (Inventory) & `stock_movements`

Tracks catalog parts and consumption logic tied to the OS lifecycle.

### Relationships Highlights

* An `OrderService` **Has Many** `OrderEquipment`, `OrderHistory`, `OrderPart`, `OrderPhoto`.
* A `Tenant` **Has One** `Subscription`, which **Belongs To** a `Plan`.
* **Universal Isolation Constraint**: The `tenantId` column is systematically integrated across structural queries generated by the backend TypeORM instances to definitively prevent data bleeding.
